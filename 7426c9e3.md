## What is lyca?

`lyca` is a **single-file**, **fastai-style** stateful `Chat` / `AsyncChat` library for
on-device **LiteRT-LM** models (Gemma 4, etc.). It is the **Lisette equivalent for
local inference** — same call-site interface, zero cloud, no HTTP server needed.

**Three hard dependencies — no others:**

| Package | Version | Role |
|---|---|---|
| `litert-lm-api-nightly` | latest | On-device LLM engine |
| `toolslm` | latest | Schema generation + tool dispatch |
| `msglm` | latest | OpenAI-format message helpers |
| `fastcore` | latest | transitive (pulled by toolslm/msglm) |

## Verified LiteRT-LM Python API (do not guess, use only what is documented here)

Install:
```bash
pip install litert-lm-api-nightly
```

### Engine + Conversation lifecycle

```python
import litert_lm

# Engine: loads model once, expensive — keep alive
with litert_lm.Engine("path/to/model.litertlm") as engine:

    # Conversation: manages turn history internally
    # messages = initial history (system + prior turns)
    # tools    = list of Python callables — LiteRT-LM generates schemas from docstrings/type hints
    with engine.create_conversation(messages=messages, tools=tools) as conv:

        # Synchronous — returns full response
        resp = conv.send_message("What is 123 + 456?")
        text = resp["content"][0]["text"]          # string

        # Streaming — iterator of chunks (synchronous iterator, not async)
        for chunk in conv.send_message_async("Tell me a long story."):
            for item in chunk.get("content", []):
                if item.get("type") == "text":
                    print(item["text"], end="", flush=True)
```

### Message format (OpenAI-compatible dicts)

```python
# System
{"role": "system", "content": [{"type": "text", "text": "..."}]}

# User (text)
{"role": "user", "content": [{"type": "text", "text": "..."}]}

# User (multimodal — audio, vision)
{"role": "user", "content": [
    {"type": "audio", "path": "/path/to/audio.wav"},
    {"type": "text", "text": "Describe this."},
]}

# Assistant (after tool call)
{"role": "assistant",
 "tool_calls": [{"function": {"name": "fn", "arguments": {...}}}],
 "tool_responses": [{"name": "fn", "response": result}]}
```

### Native tool calling (LiteRT-LM handles dispatch internally)

```python
def add_numbers(a: float, b: float) -> float:
    """Adds two numbers.
    Args:
        a: The first number.
        b: The second number.
    """
    return a + b

with engine.create_conversation(tools=[add_numbers]) as conv:
    resp = conv.send_message("What is 123 + 456?")
    print(resp["content"][0]["text"])  # "579"
```

LiteRT-LM reads type hints + Google-style docstrings to generate schemas.
The model calls the function automatically; `send_message` returns the final answer.

### Gemma 4 raw tool-call token format (when NOT using native dispatch)

When tools are passed as JSON schemas (not callables), the model emits:
```
<|tool_call>call:fn_name{arg1:<|"|>value<|"|>,arg2:42}<tool_call|>
```

Reference parser from official Gemma 4 docs:
```python
import re
def extract_tool_calls(text):
    def cast(v):
        try: return int(v)
        except:
            try: return float(v)
            except: return {'true': True, 'false': False}.get(v.lower(), v.strip("'\""))
    return [{"name": n, "arguments": {
                k: cast((v1 or v2).strip())
                for k, v1, v2 in re.findall(r'(\w+):(?:<\|"\|>(.*?)<\|"\|>|([^,}]*))', args)
             }}
            for n, args in re.findall(r"<\|tool_call>call:(\w+)\{(.*?)\}<tool_call\|>", text, re.DOTALL)]
```

```python
from toolslm.funccall import get_schema, call_func, call_func_async, mk_ns

# get_schema: Python function → JSON schema dict
# Returns: {"name": "fn", "description": "...", "input_schema": {...}}

schema = get_schema(f, pname="input_schema")

result = call_func("fn_name", {"arg": "val"}, ns)
result = await call_func_async("fn_name", {"arg": "val"}, ns)

# mk_ns: list of callables → {name: fn} dict
ns = mk_ns([fn1, fn2])
```

## The Design

### Single file: `lyca.py`

Everything in one file. No submodules. nbdev-style `#|` export markers.

### Exports

```
__all__ = ['Response', 'Chat', 'AsyncChat', 'get_text', 'extract_tool_calls']
```

### `Response` — return value of `Chat.__call__`

A **dataclass** (not NamedTuple, not class with __init__). Fields:
- `content: str` — assistant's text response
- `tool_calls: list` — list of parsed tool call dicts (may be empty)  
- `finish_reason: str = "stop"` — `"stop"` | `"tool_calls"`
- `model: str = ""`

Must implement `_repr_markdown_` for Jupyter display (mirrors Lisette's `ModelResponse._repr_markdown_`).

### `Chat` — primary class

**Long-lived engine + conversation** pattern:
- `Engine` is created once in `__init__` (expensive, ~3s on first call)
- `Conversation` is created once with initial system + history
- `send_message` is called on each `__call__`
- Both engine and conversation are context managers; `Chat` wraps them

**Constructor signature** (mirrors Lisette's `Chat.__init__`):
```python
def __init__(
    self,
    model_path: str,          # Path to .litertlm file
    sp: str = "",             # System prompt
    tools: list = None,       # Python callables (passed to create_conversation)
    hist: list = None,        # Pre-populated history (strings OR dicts)
    max_steps: int = 5,       # Max tool-call rounds (informational; LiteRT handles natively)
    max_tokens: int = 1024,   # Not directly exposed in Python API — reserved
):
```

**`__call__` signature** (mirrors Lisette's `Chat.__call__`):
```python
def __call__(self, pr=None) -> Response:
    "Send `pr` to model; return Response. LiteRT-LM dispatches tools natively."
```

**Resource management** — `Chat` must be usable both ways:
```python
# Context manager (preferred)
with Chat("model.litertlm") as c:
    r = c("hello")

# Imperative (also supported)
c = Chat("model.litertlm")
r = c("hello")
c.close()  # explicit cleanup
```

Implement `__enter__`/`__exit__` + `close()`. Use `atexit.register(self.close)` as fallback.

**History tracking**: LiteRT-LM conversations do NOT expose history. Maintain `self.hist: list[dict]` internally so sessions can be saved/restored.

**`reset()` method**: Clear `self.hist` and recreate the conversation (keeps engine alive).

**`h` property**: shorthand for `self.hist` — mirrors Lisette.

### `AsyncChat` — async variant

Inherits from `Chat`. Overrides `__call__` to return an `AsyncGenerator` that:
1. Yields `str` chunks (tokens from `send_message_async`)
2. Finally yields a `Response` on completion

```python
async for chunk in chat("hello"):
    if isinstance(chunk, str): print(chunk, end="")
    else: r = chunk  # Response
```

Wraps `send_message_async` (which is a synchronous iterator!) via `asyncio.get_event_loop().run_in_executor()`. This avoids blocking the event loop.

Also provide `acall(pr) -> Response` convenience method (collects all, returns final Response).

### Helper functions (module-level)

```python
def get_text(response: dict) -> str:
    "Extract text string from a LiteRT-LM response dict."

def extract_tool_calls(text: str) -> list[dict]:
    "Parse Gemma 4 <|tool_call>...</tool_call|> tokens from raw text."
    # Use the official regex from https://ai.google.dev/gemma/docs/capabilities/text/function-calling-gemma4
```

### Tool schema conversion

toolslm's `get_schema` returns `{"name", "description", "input_schema"}`.
LiteRT-LM native tool calling accepts raw Python callables directly and generates its own schemas. **Pass callables directly** — do not convert schemas.

For cases where the user wants to see what schema would be generated, provide:
```python
def tool_schema(f) -> dict:
    "Show toolslm JSON schema for function `f` (informational — lyca passes callables directly)."
    return get_schema(f, pname="parameters")
```

## Usage Examples (implement these exactly, they are acceptance tests)

### 1. Basic chat
```python
from lyca import Chat

c = Chat("./models/gemma4-e4b/gemma-4-e4b-it.litertlm", sp="Be terse.")
r = c("What is SQLite FTS5?")
print(r.content)
```

### 2. Stateful multi-turn
```python
c = Chat("./models/gemma4-e4b/gemma-4-e4b-it.litertlm")
c("My name is Karthik.")
r = c("What's my name?")
assert "Karthik" in r.content
```

### 3. Tool calling
```python
from lyca import Chat

def search(query: str, top_k: int = 5) -> list:
    """Search the knowledge base.
    Args:
        query: The search string.
        top_k: Number of results to return.
    """
    return [{"content": f"Result for {query}"}]

c = Chat("./models/gemma4-e4b/gemma-4-e4b-it.litertlm", tools=[search])
r = c("Search for 'RRF scoring' and summarise.")
print(r.content)
```

### 4. Streaming (AsyncChat)
```python
import asyncio
from lyca import AsyncChat

async def main():
    async with AsyncChat("./models/gemma4-e4b/gemma-4-e4b-it.litertlm") as c:
        async for chunk in c("Write a haiku about SQLite."):
            if isinstance(chunk, str): print(chunk, end="", flush=True)

asyncio.run(main())
```

### 5. Pre-populated history
```python
from lyca import Chat

c = Chat(
    "./models/gemma4-e4b/gemma-4-e4b-it.litertlm",
    hist=["What is 2+2?", "4", "And 3+3?"]  # alternating user/assistant
)
r = c("And 4+4?")  # model has full context
```

## Style Rules (fastai-style — enforce strictly)

1. **No superfluous comments** — code is self-documenting via function names + docstrings
2. **`store_attr()` replaces all manual `self.x = x`** assignments
3. **`listify()`** normalises `None`/single/list inputs
4. **`L()`** instead of `list()` when functional methods are used
5. **`@delegates`** for `AsyncChat.__init__` — forward to `Chat.__init__`
6. **No `if __name__ == "__main__"`** block  
7. **`# %% ` cell markers** for logical sections (nbdev style)
8. **Docstrings on all exported symbols** — first line is the description
9. **`@patch`** to add methods to classes post-definition (where it reduces coupling)
10. **No `Optional[X]`** — use `X = None` default

## Constraints

- Do **not** implement a prompt template builder — LiteRT-LM handles this internally
- Do **not** re-implement `mk_msg`/`mk_msgs` — import from `msglm`
- Do **not** re-implement `get_schema`/`call_func`/`mk_ns` — import from `toolslm.funccall`
- Do **not** add `ucall`, `fastapi`, `karma`, `litesearch` imports — those are caller-level concerns
- `litert_lm.Engine` is expensive to init; **never create more than one per model path** — use a module-level `_engines: dict[str, Engine]` cache identical to the singleton pattern
- `Conversation` is cheap to create — it is OK to recreate on `reset()`
- History format in `self.hist` must be the **LiteRT-LM native dict format**, not OpenAI format — check the documented message shapes above
